# API Contract v1

Versioned spec for the OpenClaw bridge HTTP API. All endpoints prefixed `/v1/`. Breaking changes ship as `/v2/` alongside `/v1/` until clients migrate.

## Transport

- HTTP/1.1 + JSON, no streaming in v1 (LLM streaming deferred to v1.1)
- Listener: `127.0.0.1:8788` (loopback only)
- TLS: not used on loopback in v1. Defence-in-depth via auth + scopes.
- Content-Type: `application/json` for all request and response bodies
- Encoding: UTF-8 only

## Authentication

Bearer token in `Authorization: Bearer <token>` header on every request except `GET /v1/health`.

Tokens are generated by the bridge admin CLI and stored in macOS Keychain on the bridge side. Each component (relay, brain, scheduler, ad-hoc CLI) gets its own token, bound to:

- An `actor` identifier (e.g. `relay.clu`, `brain.tron`, `cli.giuseppelopes`)
- A set of scopes from the catalogue below
- Optional rate-limit overrides

Token rotation is a single CLI command. Old tokens stay valid for a 24h grace window.

### Scope catalogue

| scope | grants |
|---|---|
| `imessage:send` | `POST /v1/imessage/send` |
| `imessage:relay` | `POST /v1/imessage/inbound` |
| `apple:calendar:read` | `GET /v1/calendar/*` |
| `apple:calendar:write` | `POST /v1/calendar/*`, `PATCH`, `DELETE` |
| `apple:reminders:read` | `GET /v1/reminders` |
| `apple:reminders:write` | `POST/PATCH/DELETE /v1/reminders` |
| `apple:contacts:read` | `GET /v1/contacts/*` |
| `email:read` | `GET /v1/email/*` |
| `email:send` | `POST /v1/email/send` |
| `vault:read` | `GET /v1/vault/*` |
| `vault:write` | `POST /v1/vault/*` |
| `llm:call` | `POST /v1/llm/complete` |
| `events:publish` | `POST /v1/events/publish` |
| `events:subscribe` | `GET /v1/events/subscribe` (WebSocket) |
| `admin` | scope and token management endpoints |

A request to a scope the actor doesn't hold returns `403 forbidden` with code `forbidden_scope`.

## Standard headers

Every request:
- `Authorization: Bearer <token>` (required except health)
- `X-Request-ID: <uuid>` (optional; bridge generates one if missing and echoes it back)
- `Idempotency-Key: <uuid>` (optional on POST; mandatory in v1.1)

Every response:
- `X-Request-ID: <uuid>` (always, echoed or generated)
- `X-Bridge-Version: 1.0.0`

## Error envelope

Every non-2xx response has this body:

```json
{
  "error": {
    "code": "snake_case_identifier",
    "message": "Human-readable explanation.",
    "details": {},
    "request_id": "uuid"
  }
}
```

### Standard error codes

| HTTP | code | meaning |
|---|---|---|
| 400 | `bad_request` | malformed body, missing field, invalid enum |
| 401 | `unauthorized` | missing or invalid token |
| 403 | `forbidden_scope` | token valid but lacks scope |
| 404 | `not_found` | resource id does not exist |
| 409 | `conflict` | state precondition failed (e.g. write to existing path with mode=create) |
| 409 | `idempotency_replay` | same key, different body |
| 422 | `validation_failed` | structural OK but semantically invalid |
| 429 | `rate_limited` | scope or actor bucket exhausted; `Retry-After` header set |
| 502 | `dependency_unavailable` | downstream (Apple bridge, IMAP, Redis, LLM) unreachable |
| 500 | `internal_error` | bug; full stack in bridge logs under `request_id` |

## Idempotency

POST endpoints accept `Idempotency-Key`. Bridge stores `(key, body_hash) → response` for 24h.

- Same key, same body hash → returns cached response with `X-Idempotency-Replay: true`
- Same key, different body hash → `409 idempotency_replay`
- No key supplied → request executes normally (v1; tightens in v1.1)

## Rate limiting

Token bucket per `(actor, scope)`. Defaults:

- `llm:call`: 60 req/min, burst 10
- `imessage:send`: 30 req/min, burst 5 (also throttled at the relay layer to dodge Apple anti-spam)
- `vault:write`: 120 req/min, burst 20
- everything else: 300 req/min, burst 50

Exceeded → `429` with `Retry-After` header in seconds.

## Endpoint catalogue

### Health

`GET /v1/health` — no auth.

```json
{
  "status": "ok",
  "version": "1.0.0",
  "uptime_s": 12345,
  "deps": {
    "redis": "ok",
    "apple_bridge": "ok",
    "imap_glysk": "ok",
    "imap_lopes": "ok",
    "imap_whilesum": "ok",
    "openrouter": "ok"
  }
}
```

Dep status is one of `ok | degraded | down`. Bridge stays `ok` even if non-critical deps are degraded.

### iMessage

`POST /v1/imessage/send` — scope `imessage:send`

```json
// request
{ "from": "clu|tron|flynn|main", "to": "+39...", "body": "string", "service": "iMessage|SMS" }
// response 202
{ "message_id": "uuid", "queued_at": "iso8601" }
```

Async. Bridge enqueues a job for the relay running as that user. Confirmation arrives via `imessage.sent.{agent}` event.

`POST /v1/imessage/inbound` — scope `imessage:relay`

Called by relays only. Pushes a freshly-observed inbound message into the bridge.

```json
// request
{
  "agent": "clu|tron|flynn",
  "from": "+39...",
  "body": "...",
  "received_at": "iso8601",
  "chat_guid": "..."
}
// response 200
{ "received": true, "event_id": "uuid" }
```

Side effect: publishes `imessage.received.{agent}` event.

### Calendar

`GET /v1/calendar/events?from=ISO&to=ISO&calendar=optional` — scope `apple:calendar:read`

```json
{
  "events": [
    { "id": "...", "title": "...", "start": "iso8601", "end": "iso8601",
      "calendar": "Personal|Glysk|Whilesum", "location": "...", "notes": "..." }
  ]
}
```

`POST /v1/calendar/events` — scope `apple:calendar:write`

```json
// request
{ "calendar": "...", "title": "...", "start": "iso8601", "end": "iso8601",
  "location": "...", "notes": "..." }
// response 201
{ "id": "...", "url": "calshow:..." }
```

`PATCH /v1/calendar/events/{id}`, `DELETE /v1/calendar/events/{id}` — same scope.

### Reminders

`GET /v1/reminders?list=optional&completed=false` — scope `apple:reminders:read`
`POST /v1/reminders` — scope `apple:reminders:write`
`PATCH /v1/reminders/{id}`, `DELETE /v1/reminders/{id}`

Schema mirrors EventKit.

### Contacts

`GET /v1/contacts/search?q=...` — scope `apple:contacts:read`

Returns minimal contact records: `{ name, phones[], emails[] }`. No write operations in v1.

### Email

`GET /v1/email/threads?account=glysk|lopes|whilesum&query=...&limit=20&before=iso8601` — scope `email:read`
`GET /v1/email/threads/{id}` — scope `email:read`
`POST /v1/email/send` — scope `email:send`

```json
// send request
{
  "account": "glysk|lopes|whilesum",
  "to": ["..."], "cc": ["..."], "bcc": ["..."],
  "subject": "...",
  "body_text": "...",
  "body_html": "...",
  "in_reply_to": "message-id-or-null"
}
// response 202
{ "message_id": "...", "queued_at": "..." }
```

### Vault

`GET /v1/vault/read?path=relative/path.md` — scope `vault:read`

```json
{ "path": "...", "content": "...", "frontmatter": {}, "size": 1234, "modified_at": "..." }
```

`POST /v1/vault/write` — scope `vault:write`

```json
// request
{
  "path": "01 - Projects/...",
  "mode": "create|append|replace",
  "content": "string (markdown body, no frontmatter)",
  "frontmatter": { "created": "...", "topic": "...", "status": "..." }
}
// response 201 (create) or 200 (append/replace)
{ "path": "...", "size": 1234, "written_at": "..." }
```

`mode=create` returns `409 conflict` if path exists. `mode=replace` requires existing file. `mode=append` creates if missing.

### LLM

`POST /v1/llm/complete` — scope `llm:call`

```json
// request
{
  "task_class": "triage|classify|reason|draft|summarise",
  "provider_hint": "auto|local|openrouter",
  "model_hint": "optional model id",
  "messages": [ { "role": "system|user|assistant", "content": "..." } ],
  "max_tokens": 1024,
  "temperature": 0.2,
  "response_format": "text|json"
}
// response 200
{
  "provider": "openrouter",
  "model": "anthropic/claude-haiku-4.5",
  "content": "...",
  "usage": {
    "prompt_tokens": 412,
    "completion_tokens": 89,
    "cost_usd": 0.00031
  },
  "latency_ms": 842
}
```

Routing logic (v1):
- `provider_hint=auto` → routes by `task_class`. `triage|classify` prefer local if available, else openrouter; `reason|draft|summarise` always openrouter.
- Local provider is unavailable until enabled in config — until then everything routes to openrouter.
- Telemetry written automatically (see Telemetry Plan).

### Events

`POST /v1/events/publish` — scope `events:publish`

```json
// request
{ "topic": "agent.tron.draft.pending", "payload": {...}, "ttl_s": 60 }
// response 202
{ "event_id": "uuid", "published_at": "..." }
```

`GET /v1/events/subscribe?topic=pattern` — scope `events:subscribe`, WebSocket upgrade

Topic pattern supports `*` wildcard at any segment (`imessage.received.*`). Server pushes events as JSON frames using the envelope from `Event Bus.md`.

## Versioning policy

- Path-versioned (`/v1/`, `/v2/`). Two majors run in parallel for a minimum of 30 days.
- Additive changes (new optional fields, new endpoints, new error codes) ship in-place without a major bump and are documented as `v1.x` in `X-Bridge-Version`.
- Breaking changes (field removal, type change, semantic change) require a new major.
- The OpenAPI spec generated by FastAPI is committed to the repo at `docs/openapi-v1.yaml` and is the source of truth for the typed `brains/shared` SDK.

## Amendments — 2026-04-29 (Session 1)

These supersede the corresponding parts of the spec above. Reasons documented in `SESSION-NOTES.md` from Session 1.

### Unhandled-exception handler lives in middleware

The bridge does **not** register a FastAPI `Exception` / `500` handler via `add_exception_handler`. Doing so routes errors through Starlette's `ServerErrorMiddleware`, which sits *outside* user middleware and writes its response without the `X-Request-ID` / `X-Bridge-Version` headers — breaking the spec invariant that those headers are present on every response.

Instead, the outermost user middleware (`RequestIDMiddleware`) catches unhandled exceptions itself and emits the standard error envelope from there. This guarantees header stamping on 500 responses.

### Plain ASGI middlewares, not `BaseHTTPMiddleware`

Same root cause. `BaseHTTPMiddleware`'s anyio-based exception handling short-circuits its post-response code on exceptions, which would also break header stamping. The bridge uses plain ASGI middlewares wrapping `send` — clean and exception-safe.

### Integer HTTP status literals are canonical

This document already uses integer status codes (200, 401, 409, etc.). That is intentional and now explicit policy: do **not** use `fastapi.status.HTTP_*_*` constants. Under starlette 1.0 several constants emit deprecation warnings (e.g. `HTTP_422_UNPROCESSABLE_ENTITY` was renamed per RFC 9110). With `filterwarnings = ["error"]` in pytest config, those warnings break test runs. Integer literals are stable and self-documenting.

## Amendments — 2026-04-29 (Session 2)

These supersede the corresponding parts of the spec above. Reasons documented in the Session 2 entry of `SESSION-NOTES.md`.

### Tokens live in macOS Keychain (Session 1 JSON store deprecated)

Tokens are written to one Keychain item per actor under service `com.giuseppelopesme.openclaw.bridge`. Item password is JSON `{token, previous_token, previous_expires_at, scopes}`. Enumeration is via a manifest entry (`_actors_`) the bridge maintains itself; `keyring` has no portable list-by-service API.

The bridge builds a `sha256(token) → (actor, scopes)` lookup map at startup and refreshes lazily on a 60s TTL. CLI tools that mint or rotate tokens write to Keychain; the running bridge picks up the change within one minute, or immediately on restart.

The `~/.openclaw/tokens.dev.json` fallback path is preserved for one transitional revision: empty Keychain + present JSON file ⇒ bridge reads the JSON and emits a structured `token_store_fallback_to_json` warning. Removed in Session 3 once Keychain is verified in real use.

### `Retry-After` is a real response header on 429

Per the rate-limiting section, exhausted buckets return `429 rate_limited` with `Retry-After` in seconds. As of Session 2 this header is on the response itself; previously it was implied by `details.retry_after_s` only. `details.retry_after_s` remains for callers that prefer the body field.

### Idempotency caches only 2xx responses

The 24h `(key, body_hash) → response` cache only stores successful responses. A 4xx/5xx replay re-executes — error responses are not cached. This is additive to the spec: replays of successful POSTs continue to behave as documented; failed POSTs no longer get a "stuck" replay if the underlying problem self-heals.

### Vault `mode=append` semantics

Append creates the file if missing (per the spec) and inserts a single leading `\n` when the existing file lacks a trailing newline, so consecutive appends do not run together. This is a clarification of the "creates if missing" line, not a change.

## Amendments — 2026-04-29 (Session 3)

These supersede the corresponding parts of the spec above. Reasons documented in the Session 3 entry of `SESSION-NOTES.md`.

### `/v1/health` deps map gains real probes

The `deps` object listed in the original endpoint catalogue is additive: the original keys (`redis`, `apple_bridge`, `imap_glysk`, `imap_lopes`, `imap_whilesum`, `openrouter`) are unchanged, and Session 3 adds four real-probe keys:

- `keychain` — calls into macOS Keychain through the same wrapper the bridge uses for tokens
- `vault` — `OBSIDIAN_VAULT` exists, is a directory, is listable
- `idempotency_db` — `SELECT 1` against the SQLite file
- `telemetry_db` — same against `telemetry.db`

`openrouter` is now a live probe (`GET /api/v1/models`, 2s timeout) instead of a stub.

Criticality: a critical dep being `degraded` or `down` propagates to the overall `status`. Critical = `keychain`, `vault`, `idempotency_db`, `telemetry_db`. `openrouter` and the still-stubbed keys are non-critical (the LLM endpoint owns its own errors; health shouldn't flap on a slow OpenRouter).

### LLM error responses fold timeouts into 502

OpenRouter HTTP errors and timeouts both surface as `502 dependency_unavailable`. The error envelope's `details` field carries:

- `timeout: bool` — `true` for read/connect timeouts, `false` for HTTP errors
- `upstream_status: int | null` — the upstream HTTP status when present, `null` for transport-layer failures
- `latency_ms: int` — measured at the bridge
- `model: string` — the model the bridge attempted

This avoids introducing a separate `504` status code (which would be a v1.x bump on the error catalogue).

### LLM cost is a snapshot at recording time

`usage.cost_usd` in the response and the `cost_usd` column in `telemetry.db` are computed from a hardcoded price table at recording time. The telemetry plan calls for refresh on manual cadence; historical rows are not backfilled when the table changes. Unknown models record `cost_usd=0.0`.

### LLM telemetry write asymmetry

Successful `POST /v1/llm/complete` calls write the telemetry row via FastAPI `BackgroundTasks` (after the response is sent). Failed calls write inline before the exception propagates, because Starlette skips `BackgroundTasks` attached to exception paths. Both paths use the same `write_llm_call(conn, record)` and the same swallow-on-error policy. Caller-observable behaviour (response timing, body, headers) is unchanged.

### Token store is Keychain-only

The `~/.openclaw/tokens.dev.json` fallback documented in the Session 2 amendments was removed in Session 3. Tokens live exclusively in macOS Keychain under service `com.giuseppelopesme.openclaw.bridge`. Mint with `scripts/mint-token.py` on a fresh host. The `BRIDGE_TOKEN_STORE` env var is gone.
